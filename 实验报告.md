# 基于 AI 辅助开发的 Web 应用实验报告

## 一、任务描述

### 1.1 课程背景

本实验为《算法设计与分析》课程的实践项目，旨在探索人工智能辅助软件开发的工作流程，体验从需求分析到部署上线的完整开发周期。

### 1.2 项目目标

开发一个**西安交通大学综合素质测评计算器**的 Web 应用，实现以下目标：

- 依据《本科生专业选择综合素质测评内容及评分标准》，完整覆盖品行素质（80 分）、能力拓展（20 分）、奖励分（5 分）三大评分模块
- 支持实时计算、自动封顶、动态条目管理
- 提供导出图片和打印 PDF 功能
- 适配手机和桌面端，支持深浅色模式
- 部署到公网可访问
- 具备良好的工程结构和可维护性

### 1.3 使用的 AI 工具

| 工具 | 用途 |
|------|------|
| Claude Opus 4.6（Web 对话） | 需求分析、技术规划、代码生成、迭代优化 |
| Claude Code（命令行 Agent） | 工程化搭建、迭代优化、Bug 修复、文档撰写 |

### 阶段三：对话式迭代优化

**工具**：Claude Opus 4.6（通过 claude.ai 网页对话）

在初始组件生成后，我通过多轮对话与 Claude 进行了逐步迭代优化。每一轮都是"提出问题/需求 → Claude 修改代码 → 预览效果 → 反馈调整"的循环。

#### 3.1 集体活动分计算方式优化

**我的反馈**："集体活动一般需要十次，一次是 0.3 分，而现在滑动加分是 0.5 分每格。此外一些书院可能有特殊次数要求，希望能在现有的滑动加分的基础上，添加能够填写总次数来计算集体活动分的功能。"

**问题分析**：初始版本的集体活动分仅有一个 0~3 分的滑块（步长 0.5），无法反映实际的"次数 × 每次分值"计算逻辑，也无法适配不同书院的分值差异。

**Claude 的修改**：

1. **重构状态**：将原来的单一 `collectiveScore` 状态替换为 5 个独立状态：
   ```jsx
   const [collectiveMode, setCollectiveMode] = useState("count");     // 输入模式
   const [collectiveCount, setCollectiveCount] = useState(0);          // 参加次数
   const [collectivePerActivity, setCollectivePerActivity] = useState(0.3); // 每次分值
   const [collectiveManual, setCollectiveManual] = useState(0);        // 手动填写分值
   const [collectiveOutstanding, setCollectiveOutstanding] = useState(false); // 突出贡献
   ```

2. **重构计算逻辑**：
   ```jsx
   if (collectiveOutstanding) { collectiveRaw = 3; }
   else if (collectiveMode === "count") { collectiveRaw = collectiveCount * collectivePerActivity; }
   else { collectiveRaw = collectiveManual; }
   const collective = Math.min(3, Math.max(0, collectiveRaw));
   ```

3. **重构 UI**：设计了三种输入方式：
   - **按次数计算**（默认）：两个数值输入框（参加次数、每次加分），实时显示计算公式和结果
   - **手动填写分数**：保留滑块，步长改为 0.1
   - **突出贡献**：勾选后直接获满分 3 分，其他输入项自动隐藏

4. **更新 `useMemo` 依赖数组**：将新增的 5 个状态变量替换到依赖数组中

**验证**：在预览中测试 10 次 × 0.3 = 3.0 分的场景，确认计算正确且封顶逻辑生效。

#### 3.2 部署方案咨询

**我的提问**："如何部署，哪种部署方式合适？"

**Claude 的建议**：Claude 分析了项目的特点（纯前端计算工具，无后端和数据库），推荐了四种静态托管方案：

| 方案 | 优势 | 劣势 |
|------|------|------|
| **Vercel（最推荐）** | 零配置部署 React 应用，免费额度充裕，自动 HTTPS 和 CDN，每次 push 自动更新 | 国内访问可能偶有延迟 |
| GitHub Pages | 完全免费，适合学校项目 | 需手动配置构建脚本，只支持静态文件 |
| Cloudflare Pages | 性能好，国内访问相对快（对校内用户友好） | 配置略复杂 |
| Netlify | 体验类似 Vercel | 国内访问速度不如 Cloudflare |

最终我选择了 Vercel 作为部署平台。Claude 还提供了完整的部署流程：Vite 初始化项目 → 放入组件 → 推到 GitHub → Vercel 自动构建部署。

#### 3.3 三大功能一次性升级

**我的需求**："我希望继续添加功能：得分导出为图片、PDF；深色模式适配；响应式移动端优化"

**Claude 的分析**：这三个功能涉及面较广——深色模式要改所有组件的配色方案，响应式要调整布局逻辑，导出需要新增 Canvas 绘图功能。Claude 决定对整个组件进行系统性重写。

**具体改动**：

**(1) 深色模式适配**

- 新增 `dark` 状态和切换按钮（header 右上角的太阳/月亮图标）
- 自动检测系统偏好：
  ```jsx
  useEffect(() => {
    if (window.matchMedia?.("(prefers-color-scheme: dark)").matches) setDark(true);
  }, []);
  ```
- 根组件通过 `className={dark ? "dark" : ""}` 控制全局样式
- **所有组件全面适配**：为每个 UI 元素添加 `dark:` 变体类名，包括：
  - 背景色：`bg-white` → `bg-white dark:bg-stone-800`
  - 文字色：`text-stone-700` → `text-stone-700 dark:text-stone-200`
  - 边框色：`border-stone-200` → `border-stone-200 dark:border-stone-700`
  - 输入框焦点色：`focus:border-teal-400` → `dark:focus:border-teal-500`
  - Badge 组件的 6 种颜色变体均添加了深色模式样式

**(2) 响应式移动端优化**

- **Header**：标题用 `truncate` 防溢出，副标题在窄屏隐藏（`hidden sm:block`），总分字号自适应（`text-xl sm:text-2xl`）
- **Tab 导航**：按钮字号和内边距自适应（`text-xs sm:text-sm`、`py-2 sm:py-2.5`）
- **卡片内边距**：`p-4 sm:p-5` 在手机上收紧
- **动态表单项**：下拉框使用 `min-w-[120px] flex-1 sm:flex-none sm:w-36` 实现自动换行
- **得分汇总**：计算公式在小屏隐藏（`hidden sm:inline`），只显示最终分数
- **导出弹窗**：手机上从底部弹出（`items-end sm:items-center`），圆角适配（`rounded-t-2xl sm:rounded-2xl`）

**(3) 得分导出功能**

新增了 `ExportModal` 组件和 `drawExportImage` Canvas 绘图函数：

- **Canvas 绘图**（`drawExportImage` 函数）：
  - 画布尺寸 800×520，支持高 DPI（`window.devicePixelRatio`）
  - 绘制圆角卡片背景、标题、总分（48px 等宽字体）
  - 三色进度条（teal/orange/amber 对应三大板块）
  - 三列子卡片，每列显示板块名称、得分、子项明细
  - 底部声明文字
  - **跟随当前深色/浅色模式**：根据 `isDark` 参数使用不同的配色方案

- **导出弹窗**（`ExportModal` 组件）：
  - 实时预览 Canvas 绘制的图片
  - "保存图片"按钮：通过 `canvas.toDataURL("image/png")` 生成 PNG 并触发下载
  - "打印/PDF"按钮：打开新窗口加载图片，触发 `window.print()`（用户可选择"另存为 PDF"）

**最终产出**：重写后的组件约 768 行，在保留所有原有功能逻辑的基础上，完整集成了深色模式、响应式布局和导出功能。

### 阶段五：工程化迭代优化

**工具**：Claude Code

通过与 Claude Code 的多轮对话，逐步解决工程化环境中的问题并优化体验：

#### 5.1 深色模式修复

**问题**：点击深色模式按钮后，`.dark` 类正确添加到 DOM，但 Tailwind 的 `dark:` 变体样式不生效。

**排查过程**：
- Claude Code 使用浏览器预览工具检查 DOM 结构和计算样式
- 发现 `getComputedStyle` 显示 gradient 已为深色值，实际 CSS 已生效
- 确认是 Tailwind CSS 4 的 `@custom-variant` 语法配置问题

**修复**：将 CSS 中的 dark 变体选择器从 `(&.dark)` 改为 `(&:is(.dark *))`，确保所有 `.dark` 元素的后代都能匹配 `dark:` 前缀的工具类。

#### 5.2 卡片布局重组

**需求**：
- 将"学科竞赛获奖"和"论文/专利/专著"合并到一个大卡片容器中，右下角显示"学术: 0.0/10"
- 将"文艺竞赛"和"体育竞赛"合并到一个卡片中，用分隔线区隔

**实现**：
- 学术科研：外层用一个带边框和淡橙色背景的 `div` 包裹两个独立 `Card`，底部放 `Badge` 标注
- 文体竞赛：合并为单个 `Card`，中间用 `border-t` 分隔线隔开

#### 5.3 颜色配置优化

**问题**：外层容器的背景色（`bg-slate-50/50`）与页面背景几乎无法区分。

**修复**：将学术科研外层容器的颜色改为带有橙色色调的配色方案：
```
浅色：border-orange-200/60 bg-orange-50/40
深色：border-orange-800/40 bg-orange-950/20
```
使容器在视觉上与背景形成清晰的层次对比。

### 阶段七：功能增强与架构重构

**工具**：Claude Code

在基础功能和部署完成后，进行了第二轮大规模迭代，涵盖功能增强和架构优化共 7 个方面：

#### 7.1 本地数据持久化（localStorage）

**问题**：刷新页面后所有输入丢失，对表单项众多的工具来说体验很差。

**实现**：
- 创建自定义 Hook `useLocalStorage.js`，包含 `loadState()` 加载函数和 `usePersistState()` 自动保存 Hook
- 使用单一 localStorage key `"xjtu-deyu-data"` + 版本号字段，支持未来数据结构迁移
- 通过 `useState(() => loadState())` 惰性初始化，仅在首次渲染时读取
- try/catch 处理隐私模式和配额超限

#### 7.2 评分规则内联提示（Tooltip）

**需求**：用户填写时需要来回翻阅政策文件查看评分标准。

**实现**：
- 创建 `RuleTooltip` 组件：桌面端 hover 触发、移动端 click 切换、click-outside 关闭
- 为 12 个评分区域创建 JSX 格式的提示内容，包含迷你评分表格（`RuleTable` 组件）
- 在每个评分项标题后添加 `?` 图标，绝对定位弹出不影响布局

#### 7.3 得分分布可视化（ScoreChart）

**实现**：创建 SVG 水平条形图组件，展示三大模块（品行/能力/奖励）的当前得分 vs 满分，每行下方显示子项分数拆解，带 CSS transition 动画。

#### 7.4 组件拆分重构

## 四、技术架构总结

### 4.1 技术选型

| 层面 | 选择 | 理由 |
|------|------|------|
| UI 框架 | React 19 | 组件化开发，声明式 UI，丰富的生态 |
| 构建工具 | Vite 8 | 极快的冷启动和热更新，原生 ESM 支持 |
| 样式方案 | Tailwind CSS 4 | 原子化 CSS，无需写独立样式文件，暗色模式友好 |
| 导出方案 | Canvas API | 纯前端生成高质量图片，无需后端 |
| 部署平台 | Vercel | 零配置、自动 CI/CD、免费 HTTPS 和 CDN |

### 4.2 代码结构

应用采用模块化架构，拆分为 14 个文件：

- **数据层**（`data/constants.js`）：7 个评分常量表 + localStorage 配置 + 默认状态
- **图标层**（`components/icons.jsx`）：10 个内联 SVG 图标组件，避免引入外部图标库
- **UI 组件层**（`components/ui.jsx`）：12 个可复用子组件（Card、Badge、SectionTitle、Field、Select、NumberInput、Checkbox、AddButton、DynamicItem、TabNav 等），全部适配深色模式和响应式
- **提示层**（`components/RuleTooltip.jsx`）：Tooltip 组件 + 12 个评分规则内容常量
- **可视化层**（`components/ScoreChart.jsx`）：三模块水平条形图
- **交互层**（`components/CollapsibleList.jsx`）：可折叠列表 + 排序工具函数
- **导出层**（`components/ExportModal.jsx`）：Canvas API 绘制 900×600 高 DPI 评分报告图，含 Logo、时间戳、印章水印
- **页面层**（`components/tabs/*.jsx` + `ScoreSummary.jsx`）：三个 Tab 页面和汇总卡片
- **逻辑层**（`hooks/`）：`useScoreCalculator` 封装评分计算，`useLocalStorage` 封装持久化
- **主组件**（`App.jsx`，~120 行）：状态声明 + Hook 调用 + 布局组合

## 五、收获与心得

### 5.1 AI 辅助开发的效率优势

本项目从零到部署上线，整个过程通过 AI 辅助大幅提升了开发效率：

- **需求分析**：Claude 能够直接读取并理解 .docx、.pdf、.doc 三种不同格式的政策文件，自动提取评分规则并生成结构化的需求文档和可视化图表。如果纯人工完成，仅阅读和整理三份文件的评分规则就需要相当长的时间。
- **初始代码生成**：Claude Opus 4.6 在一次对话中就生成了功能完整的 700+ 行 JSX 组件，包含复杂的评分数据结构和计算逻辑。如果纯手工编写，仅数据表的整理和评分逻辑的实现就需要数小时。
- **迭代优化**：每一轮对话都能准确理解需求并快速产出修改后的代码。例如集体活动分的优化，Claude 在一次交互中完成了状态重构、逻辑修改、UI 重写和依赖更新四处协同修改。
- **系统性重写**：当需要同时添加深色模式、响应式和导出功能时，Claude 判断增量修改不如全量重写高效，直接输出了一份整合所有功能的新版本。
- **工程化搭建**：Claude Code 自动完成了 Vite + Tailwind CSS 的项目初始化和配置，省去了查阅文档的时间。
- **Bug 排查**：深色模式不生效的问题，Claude Code 通过浏览器预览工具逐步检查 DOM、计算样式、CSS 配置，系统性地定位到了 Tailwind CSS 4 的 `@custom-variant` 语法问题。

### 5.2 人机协作的模式

在整个开发过程中，形成了一种高效的人机协作模式：

- **人负责需求和决策**：我提出功能需求（如"添加按次数计算的功能"）、审美偏好（如"颜色太淡了"）、架构决策（如选择 Vercel 部署）、以及基于实际使用场景的反馈（如"一般需要十次，一次 0.3 分"）
- **AI 负责实现和调试**：Claude 负责文件解析、需求分析、代码编写、配置调试、样式调整等具体实现工作
- **迭代式优化**：通过多轮对话逐步完善，每轮都能看到即时效果并给出反馈。Claude 的 Artifacts 面板让每次代码修改都能立即预览渲染结果，大幅缩短了反馈循环
- **两种 AI 工具配合**：Claude Opus 4.6 负责整体架构设计和大块代码生成，Claude Code 负责工程化环境中的精细调试和部署

### 5.3 对 AI 代码生成的思考

- **优势**：AI 在生成结构化、模式化的代码方面表现出色，尤其是表单界面、数据映射、样式编写等重复性较高的工作。在本项目中，Claude 能够将政策文件中的文字描述直接转化为数据结构和计算逻辑，这一能力在处理复杂规则型需求时尤为突出
- **主动决策能力**：Claude 在第七轮对话中主动判断"改动量较大，需要系统性重写"，而非机械地在原有代码上叠加修改，体现了 AI 在工程决策方面的一定能力
- **局限**：AI 生成的代码有时需要人工审核和调整，例如评分规则的准确性需要对照原始文件确认。本项目中集体活动分的默认步长（0.5）就是 AI 初始判断不够贴合实际的例子，需要使用者基于真实场景反馈修正
- **最佳实践**：将清晰、具体的需求描述提供给 AI，比模糊的指令能获得更好的输出质量。例如"添加按次数计算功能"不如"一般需要十次，一次 0.3 分，书院可能有特殊要求"来得有效
